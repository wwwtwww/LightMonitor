# 详情页 UI 布局优化建议计划

本计划旨在分析当前 LightMonitor 数据库详情页（MySQL, Oracle, SQL Server）的 UI 布局，并提出基于现代监控系统设计原则的优化建议。

## 1. 现状分析

基于对 `src/panels/*.vue` 代码的分析，目前的详情页布局结构如下：

*   **顶部操作条 (Ops Strip)**: 快速访问锁等待、慢查询等关键告警信息。
*   **KPI 卡片 (KPI Snapshot)**: 网格布局展示最新的关键指标（响应时间、QPS、连接数等）。
*   **功能标签页 (Tabs)**:
    *   **Metrics**: 趋势图表区域，使用 ECharts 展示历史数据。
    *   **Locks/Blocking**: 锁等待或阻塞会话列表。
    *   **Slow/Capacity**: 慢查询列表（MySQL）或容量监控（Oracle）。

**优点**:
*   信息结构清晰，分层明确（概览 -> 趋势 -> 详情）。
*   核心指标一目了然。
*   不同数据库面板保持了较好的一致性。

**潜在改进点**:
*   KPI 区域信息密度较高，可能缺乏分组。
*   图表区域较为单一，缺乏对比或关联分析视图。
*   静态信息（如实例版本、运行时间）展示较少。

## 2. 总体布局建议

### 2.1 头部信息增强 (Header Enhancement)
*   **建议**: 在页面顶部增加一个紧凑的“实例概览栏”。
*   **内容**: 显示数据库类型图标、实例名称/地址、版本号、运行时间 (Uptime)、当前角色 (Master/Slave) 以及一个显眼的“健康状态”指示器（Online/Offline/Warning）。
*   **目的**: 让用户进入页面瞬间获得上下文信息，不仅是动态指标。

### 2.2 KPI 卡片的分组与视觉优化
*   **建议**: 将 KPI 指标进行逻辑分组，而不是平铺。
*   **分组示例**:
    *   **性能 (Performance)**: Response Time, QPS, TPS.
    *   **资源 (Resources)**: Connections, Threads.
    *   **健康 (Health)**: Replication Lag, Slow Count, Deadlocks.
*   **视觉**: 使用微小的分割线或背景色块区分不同组；对异常值（如高延迟）使用颜色高亮（红/黄）。

### 2.3 图表区域的灵活性
*   **建议**: 引入“图表工具栏”。
*   **功能**:
    *   **时间范围选择**: 快速切换 1h, 6h, 24h, 7d（目前已有，可优化位置）。
    *   **指标切换**: 允许用户勾选/取消显示某些非核心图表，减少干扰。
    *   **全屏模式**: 允许将某个图表放大查看。

### 2.4 侧边栏/抽屉式详情 (Drawer for Details)
*   **建议**: 当点击 Locks 或 Slow Queries 列表中的某一行时，从右侧滑出详情抽屉，而不是弹窗或跳转。
*   **目的**: 保持上下文不丢失，方便用户一边看列表一边看详细 SQL 语句或执行计划（如果有）。

## 3. 针对不同数据库的特定建议

### 3.1 MySQL
*   **复制拓扑**: 如果是 Slave 节点，建议在 KPI 或图表旁增加一个可视化的“复制链路”示意图，显示 Master -> Slave 的流向及延迟。
*   **慢查询**: 慢查询列表可以增加“时间分布热力图”，直观展示慢查询发生的时间段。

### 3.2 SQL Server
*   **等待类型 (Waits)**: 目前在 KPI 中只显示 Top Wait。建议增加一个专门的“等待统计”小组件（如饼图或条形图），展示 Top 5 等待类型的比例。
*   **死锁**: 如果采集了死锁信息，可以在时间轴上用特殊标记（Annotation）标出死锁发生的时间点。

### 3.3 Oracle
*   **表空间**: 目前是列表展示。建议使用“进度条”可视化展示表空间使用率，并在达到阈值（如 90%）时变红。
*   **会话**: 可以增加 Active vs Inactive 会话的堆叠图，更直观展示系统负载。

## 4. 交互与体验优化

*   **自动刷新倒计时**: 在自动刷新按钮旁显示下次刷新的倒计时环，提供确定性。
*   **骨架屏 (Skeleton Loading)**: 在数据加载时使用骨架屏代替简单的 Loading 转圈，提升感知性能。
*   **移动端适配**: 确保 KPI 卡片在移动端自动堆叠，图表支持触摸滑动。

## 5. 下一步行动计划

如果采纳上述建议，我们可以分阶段实施：
1.  **阶段一**: 实施 KPI 分组和头部信息增强（涉及 Vue 组件结构的调整）。
2.  **阶段二**: 优化图表区域和可视化组件（如 Oracle 表空间进度条）。
3.  **阶段三**: 引入详情抽屉和高级交互（如死锁标记）。

请告知您对哪些建议最感兴趣，或者有其他特定的布局偏好。
